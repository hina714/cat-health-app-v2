# 説明・会話ルール

専門用語が出てきた場合は、初心者にもわかるように簡単な言葉で補足説明を添えること。

例：
- 「**ルーティング**（URLとページの対応付け）」
- 「**非同期処理**（時間のかかる処理を待つ仕組み）」

# 質問・会話ルール

ユーザーからの質問で主語や文脈が不明瞭な場合は、そのまま回答せず指摘すること。
その際、より明確な質問の例を提案すること。

例：
- ユーザー：「これどうすればいい？」
- AI：「主語が不明です。例えば「`src/lib/db.ts` の接続エラーはどうすればいい？」のような形で質問していただけますか？」

# データベースルール

Supabase に対して、スキーマやデータを変更するSQLを絶対に実行しないこと。
対象は CREATE、ALTER、DROP、INSERT、UPDATE、DELETE、TRUNCATE など。

スキーマ変更が必要な場合は、SQLファイルを `sql/` フォルダに作成するところまでを行い、
実際の実行はユーザーが Supabase の SQL Editor で手動で行う。

# 理解度確認ルール

ユーザーから「理解度確認」「クイズ」などを依頼された場合は、
以下の問題をチャット形式でユーザーに1問ずつ出題すること。
ユーザーが答えたら正誤と解説を伝え、次の問題に進むこと。
全問終了後、採点結果をまとめること。

## 問題

### Q1: プロジェクト構成
- `src/app/api/test/route.ts` は何をするファイルか？
- `src/lib/db.ts` は何をするファイルか？なぜグローバル変数に接続を保持しているか？

### Q2: APIの仕様
- `GET /api/test` のレスポンスの形式は？
- `POST /api/test` の必須パラメータは何か？不足していた場合のレスポンスは？
- `POST /api/test` が成功したときのHTTPステータスコードは？

### Q3: DB接続の仕組み
- DB接続に使っているライブラリは何か？
- 接続文字列はどこから取得しているか？
- 開発環境でグローバルに接続を保持している理由は何か？

### Q4: ルール・制約の理解
- Supabaseに対して絶対にやってはいけない操作は何か？スキーマ変更が必要なときはどうするか？
- QAレポートはどこに・どんなファイル名で保存するか？
- 専門用語が出てきたときにすべき補足説明とはどういうものか？

# QAルール

ユーザーから「QA」「確認テスト」「動作確認」などを依頼された場合は、以下の手順で実施すること。

## 手順

1. **実装ファイルを読む**
   - `src/app/api/` 以下のすべてのルートファイルを確認する
   - `src/lib/db.ts` を確認する

2. **APIの動作テストをする**
   - `http://localhost:3000` が起動中かどうかを確認する
   - 起動中であれば、各エンドポイントに対してリクエストを送り動作確認する
   - 起動していなければ、コードレビューのみ実施する

3. **レポートを作成する**
   - 今日の日付と連番でファイル名を決める（例: `qa/20260323_001_report.md`）
   - `qa/` フォルダ内の既存ファイルを確認して連番が重複しないようにする
   - 以下の形式でレポートを書く

## レポートの形式

```markdown
# QAレポート YYYY-MM-DD

## 確認した実装
- 確認したファイルの一覧

## テスト結果

### GET /api/xxx
- リクエスト: ...
- レスポンス: ...
- 結果: ✅ 正常 / ❌ 異常

## 問題点・改善提案

### 問題1: タイトル
- 対象ファイル: ...
- 内容: ...
- 提案: ...

## 総評
```
