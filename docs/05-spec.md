# アプリ仕様書

## 背景・課題

### 背景

実家の猫が元気をなくし便秘が続いたとき、普段は姉が病院に連れて行っていたが急遽自分が連れて行くことになった。獣医師に「いつ頃から食欲が落ちましたか？」と聞かれたとき、はっきり答えられず焦った。事前に姉に状況を確認しておくべきだったと反省すると同時に、そもそも家族全員が日頃から記録を共有できていればこういった事態は防げたと感じた。ちょうど勉強していた知識を活かして、この問題を解決できないかと考えたのがアプリ開発のきっかけとなった。

### 解決したかった課題

- 食事量・排便・体調を継続的に記録する手段がなかった
- 家族間でリアルタイムに情報を共有できなかった

---

## 目的

- 猫の日々の健康状態を記録・管理できる
- 家族間でリアルタイムにデータを共有できる

---

## スコープ

- 対象は自分の家族のみ（複数家族の管理は対象外）

---

## 機能

### アカウント登録 ✅

ユーザー名・パスワードで新規アカウントを作成する。

家族のみが使用する招待制のアプリのため、アカウント登録のAPIは設けず、ターミナルでスクリプトを実行することで行う。

### ログイン ✅

ユーザー名とパスワードを入力してログインする。

---

### 猫のプロフィール閲覧 🚧

トップページの猫カードをタップすると、プロフィール閲覧ページ（`/cats/[id]`）へ遷移する。

**表示項目**

| 項目 | 表示内容 |
|---|---|
| アイコン | 登録画像。未登録の場合は🐱を表示 |
| 名前 | テキスト |
| 品種 | テキスト。未登録の場合は「－」を表示 |
| 誕生日 | `YYYY年MM月DD日` 形式。未登録の場合は「－」を表示 |
| 年齢 | 誕生日から「X歳Xヶ月」と計算して表示。誕生日未登録の場合は非表示 |
| 去勢・避妊 | `true` → あり / `false` → なし / `null` → 不明 |

**操作**

- アイコン部分に「アイコンを変更」ボタンを表示し、タップするとアイコン編集ページへ遷移する

---

### 猫のプロフィール編集 🚧

プロフィール閲覧ページのアイコン変更ボタンから遷移する（`/cats/[id]/edit`）。アプリ上ではアイコン写真の編集のみ可能。

- アイコン写真：ファイル選択（選択後にトリミング・ズーム調整が可能）

---

### 記録管理 🚧

猫の日々の健康状態を記録・確認する。

| 項目 | 種類 |
|---|---|
| 体重 | 数値入力（kg） |
| 食事量 | 選択（よく食べた / 普通 / 少ない / 食べなかった） |
| 排泄 | 選択（正常 / 異常 / なし） |
| 体調 | 選択（良好 / 普通 / 不調） |
| 様子・メモ | テキスト入力（自由記述） |

- 記録には**誰が書いたか**（ユーザー名）を表示する
- 記録の編集は**書いた本人のみ**可能
- 家族は記録に対して**コメント**を投稿できる

---

## データ設計

### テーブルの関係

```
cats（猫）                       ← 家族全員で共有（特定ユーザーに紐づけない）

records（健康記録）
  └── user_id → users           ← 誰が書いたか（編集権限の判定に使う）

comments（コメント）
  ├── record_id → records       ← どの記録へのコメントか
  └── user_id   → users         ← 誰が書いたか
```

> 今回は「1家族 = 1アプリ」の前提のため、猫は特定のユーザーに紐づけない。
> 全ユーザーが全ての猫を共有できるシンプルな設計とする。

---

### users

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | UUID | ✓ | 主キー（自動生成） |
| `username` | TEXT | ✓ | ユーザー名（一意） |
| `password_hash` | TEXT | ✓ | ハッシュ化されたパスワード |
| `created_at` | TIMESTAMPTZ | | 登録日時（自動セット） |

### cats

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | UUID | ✓ | 主キー（自動生成） |
| `name` | TEXT | ✓ | 猫の名前 |
| `icon_data` | TEXT | | アイコン画像（base64） |
| `breed` | TEXT | | 品種 |
| `birthdate` | DATE | | 誕生日 |
| `neutered` | BOOLEAN | | 去勢・避妊（true: あり / false: なし / null: 不明） |
| `created_at` | TIMESTAMPTZ | | 登録日時（自動セット） |

### records

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | UUID | ✓ | 主キー（自動生成） |
| `user_id` | UUID | ✓ | 書いたユーザー（`users.id`） |
| `weight` | NUMERIC | | 体重（kg） |
| `food_amount` | TEXT | | 食事量（よく食べた / 普通 / 少ない / 食べなかった） |
| `excretion` | TEXT | | 排泄（正常 / 異常 / なし） |
| `condition` | TEXT | | 体調（良好 / 普通 / 不調） |
| `memo` | TEXT | | 様子・メモ（自由記述） |
| `created_at` | TIMESTAMPTZ | | 記録日時（自動セット） |

### comments

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | UUID | ✓ | 主キー（自動生成） |
| `record_id` | UUID | ✓ | 対象の記録（`records.id`） |
| `user_id` | UUID | ✓ | 書いたユーザー（`users.id`） |
| `body` | TEXT | ✓ | コメント本文 |
| `created_at` | TIMESTAMPTZ | | 投稿日時（自動セット） |

## 技術スタック

| 用途 | 技術 |
|---|---|
| フレームワーク | Next.js |
| 言語 | JavaScript / TypeScript |
| DB | Supabase (PostgreSQL) |
| DB接続 | postgres.js |
| 認証 | JWT |

### 採用理由

**Next.js**
Reactを学習していく中でその延長としてNext.jsに興味を持ち、実際のプロジェクトで使ってみたいと考えたため採用した。

**Supabase**
学生でも無料で利用できるDBサービスであり、コストをかけずに本番相当の環境を構築できる点から採用した。

**postgres.js**
ORMを使うと裏側でどんなSQLが動いているかわからないため、最初は自分でSQLを書いて仕組みを理解したいと思い採用した。ちょうど学校でSQLを学ぶ予定もあったため、実際のプロジェクトで試せる良い機会だと考えた。

**JWT認証**
ユーザー名とパスワードによるログインに対して、認証後にJWTトークンを発行する形式を採用した。認証まわりは理解が難しい部分も多く、わからないところはエンジニアの兄やAIに質問しながら実装を進めた。またSupabaseには標準でAuthという認証機能が用意されているが、勉強を兼ねて今回は自前で実装することにした。
