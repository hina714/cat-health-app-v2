# 仕様書の書き方ガイド

## 基本の構成順序

```
1. 目的     → このアプリ・機能は何のためにあるか
2. 画面     → どんな画面があって、何ができるか
3. データ   → それを実現するためにDBはどう設計するか
```

読む人の思考の流れに合わせる。「何を作るか」が決まってから「どう保存するか」を読む順にする。

---

## 各セクションで書くこと

### 目的
- 誰のためのアプリか
- 何ができるか（1〜2文で）
- 対象外のことも書く（例：「複数家族の管理は対象外」）

### 画面
- 画面名と URL を明示する（例：`/cats/edit`）
- 入力項目はテーブルで「項目 / 種類 / 必須」をまとめる
- 「保存後にどこへ遷移するか」など動作の流れも書く
- 機能一覧と詳細は分けず、**画面ごとにまとめて書く**

### データ設計
- テーブルの関係を図で示す
- 各テーブルのカラムは「名前 / 型 / 必須 / 説明」のテーブルで書く
- なぜその設計にしたか（理由・前提）も補足する

---

## 文章を書くときの注意

| やること | やらないこと |
|---|---|
| 短く・1文1意味にする | 1文に複数の情報を詰め込む |
| 主語を明確にする | 「これ」「それ」で済ませる |
| 決まっていないことは「未定」と書く | 曖昧なまま放置する |
| 設計の理由・前提を補足する | 結論だけ書いて理由を省く |

---

## Markdown の使い方

```markdown
## 大見出し      → セクションの区切り
### 中見出し     → セクション内の小項目
| テーブル |     → 項目の一覧や比較
- リスト         → 箇条書き（順序がないとき）
**太字**         → 特に重要なキーワード
> 引用           → 補足・注意・背景説明
```

## 進捗の書き方

| 記号 | 意味 |
|---|---|
| ✅ | 完成 |
| 🚧 | 実装中・未実装 |
| ❌ | 廃止・対象外 |
